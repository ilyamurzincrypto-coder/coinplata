## Сообщение

Привет, Кирилл!

19 мая (понедельник) у нас cutover на новый ledger. Нужны актуальные
балансы по всем нашим crypto-кошелькам в двух точках:

- **T-2 (17 мая, суббота)** — предварительный snapshot (для dry-run на
  staging). Если до cutover балансы изменятся — пришлёшь новый.
- **T-0 (19 мая утром, до 10:00 UTC)** — финальный snapshot, который
  попадёт в opening transaction. После этого все новые операции через
  новый ledger; старая система фиксируется как read-only архив.

### Зачем

Cutover = единоразовый перенос текущих балансов в новый учёт. Crypto
учитывается отдельно от cash (cash приходит от менеджеров на местах,
crypto — от тебя как treasury source-of-truth).

### Формат

JSON-массив, по одной записи на каждый wallet с ненулевым балансом:

```json
[
  {
    "wallet_id": "uuid-из-ledger.wallet_addresses",
    "currency": "USDT",
    "balance": 250000.00
  },
  {
    "wallet_id": "...",
    "currency": "USDT",
    "balance": 1500000.00
  }
]
```

`wallet_id` = ID из таблицы `ledger.wallet_addresses` (я могу прислать
текущий список с label/address для маппинга — попроси).

`balance` — точное число, как в blockchain explorer. Дробная часть до
8 знаков для USDT/USDC/BTC.

`currency` — для проверки соответствия (если currency не совпадёт с
тем что в БД для этого wallet — RPC упадёт).

### Pending транзакции

Если на момент snapshot есть **исходящая** pending транзакция (мы
отправили, но ещё не подтверждена) — **вычитать из balance**, как
будто уже ушло. Это исключит double-counting при receive в новом
ledger после cutover.

Если есть **входящая** pending (нам пришло, но ещё не подтверждено)
— **не включать в balance**. Учтём как обычную операцию в новом
ledger когда подтвердится.

### Что делать с пустыми кошельками

Wallets с balance=0 пропускать. Если позже на них прилетит — операция
автоматически заведёт balance в новом ledger.

### Срок ответа

T-2 snapshot до 17 мая 18:00 UTC.
T-0 snapshot до 19 мая 10:00 UTC.

Спасибо!
