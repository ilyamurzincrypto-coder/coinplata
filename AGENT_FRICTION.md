## Раздел 1 — Диагноз (7 причин, каждая с доказательством)

### D1. Нет статических ворот: `typecheck` и `lint` отсутствуют
Проект на **чистом JS** (нет `tsconfig.json`, нет `.eslintrc`/`eslint.config`). Единственные ворота — `build` (не ловит логику) и тесты. Значит агент **физически не может** статически поймать: неиспользуемые переменные, теневые импорты, опечатку в имени поля, `undefined`-проп, неверную сигнатуру. Он узнаёт о проблеме только когда её видит человек на экране.
**Плюс:** тесты реально есть и зелёные — `vitest run` → **596 тестов / 59 файлов**. Это сильная страховка бизнес-логики, но она не покрывает UI/визуал/новые панели.
**Доказательство:** `ls tsconfig*.json .eslintrc*` → нет; `npm test` → 596 passed.

### D2. Дрейф схемы: агент вынужден гадать про БД
137 миграций в репо, но **8 боевых объектов без DDL**: `external_rates, special_rates, rapira_alert_state, v_external_rates_latest, set_pair_margins, replace_special_rates, buy_margin, coinpoint_office_code` (`grep -rl <obj> supabase/migrations` = 0 для каждого). Агент каждый раз лезет интроспектить живую БД через mcp, чтобы узнать имена полей/функций. Часть «нового» слоя курсов не воспроизводится из репо вообще.
**Следствие:** без живого доступа агент **додумывает** поля и сигнатуры (и ошибается — см. Раздел 3).

### D5. Дубли и «несколько способов одного и того же»
- **Расчёт курса — минимум 3 несогласованных места:** фронтовый `getRate` (каскад+пивот), серверный `effective_rate`/`deal_rate_for_leg` (SQL, вне репо), и **скопированный** `usdtPer` в двух панелях (`CrossRatesPanel.jsx` + `RatesAuxPanel.jsx`) — один читает глобальные курсы, другой офисные (см. BUGS B2/B3). `getRate` фигурирует в **46 файлах**.
- **4 форматтера курса** с разными порогами точности (BUGS B6).
- **Два пути создания сделки:** legacy `create_deal` и v2 `create_deal_v2` за флагом `VITE_USE_NEW_LEDGER`; две формы сделки — legacy `ExchangeForm` и v2 `DealForm` (флаг `USE_NEW_DEAL_FORM` зашит false).
Агент не знает, какая реализация «настоящая», и легко правит не ту / дублирует.

### D6. Неявная, недокументированная конвенция ориентации курса
`getRate` возвращает **«читаемое» значение >1** (напр. TRY→USDT хранится как 46.8, а не 0.021), и вызывающий обязан инвертировать по правилу «STRONG={USD,EUR} → как есть, иначе 1/x; percent-пары → отдельно». Это **нигде не описано**. Агент, не зная этого, умножил напрямую и получил **1 TRY = 3631 RUB** (см. Раздел 3, ошибка №2). Это же правило — источник бага B2 (GBP/CHF).

### D7. Контекст-раздувание
`translations.jsx` — **4589 строк** одного `DICT` (en/ru/tr). Любое касание i18n тянет огромный файл в контекст и вытесняет полезное. Плюс в корне копятся крупные `.md` (RATES_MAP, RATES_MIGRATION_PLAN, STATUS_REPORT…), которые агент подхватывает.

## Раздел 3 — Самоанализ: где я (агент) ошибался в этом репозитории и почему

Честно, по фактам из ЭТОЙ сессии.

**1. Делал без проверки видимого результата.** Я гоняю `build` (проходит) и полагаю «работает», но UI не вижу — только твои скрины с лагом. Отсюда серия правок «показал/спрятал/подвинул» (тренд «было», щель дровера, «=» вплотную к спреду). *Чего не хватило:* способа увидеть отрендеренную панель. *Что помогло бы:* сразу давать 1 скрин + словами «что не так», а мне — не отгружать визуальные вещи без явной проверки.

**2. Додумывал математику вместо чтения существующего.** Перестановки: `1 TRY = 3631 RUB` — я умножил `getRate("TRY","USDT")` напрямую, не посмотрев, что в кассе `getRate` отдаёт **читаемое >1** и что рядом уже есть `usdtPer` с правильной ориентацией (`CrossRatesPanel.jsx`). *Выстрелило:* курс в 2000 раз мимо. *Хуже:* когда чинил — **скопировал** `usdtPer` в новый файл вместо импорта → создал дубль B3. *Корень:* не прочитал соседний хелпер (D6, D5).

**3. Не посмотрел на источник истины перед маппингом.** Tolunay: показывал **среднее (46,60)** для обоих направлений, пока ты не ткнул в саму страницу с «Покупка/Продажа». Надо было сразу открыть `tolunaylar.com.tr` и увидеть bid/ask.

**4. Отгрузил ложные стрелки.** Тренд «было»: текущая цена бралась из фолбэка (глобальная пара), а «было» — из истории Tolunay; сравнивал **разные источники** → ложное ▼. Не проверил, что обе стороны сравнения из одного источника, прежде чем пушить.

**5. Разрастание скоупа / лишние допущения.** Добавил фильтр `active !== false` в редакторе курсов — ты справедливо: «это настройки, нахуя фильтровать». Я перенёс логику «показывай активные» из другого контекста, не спросив. Аналогично захардкодил «РФ/Турция», не подумав про масштаб, — пришлось обобщать.

**6. Ошибка ветвления при деплое.** Коммит для coinpoint сначала черри-пикнул на **устаревшую** ветку (60 коммитов позади), заметил только по расхождению — переделал через worktree `main`. Не проверил ветку до коммита.

**Что конкретно нужно от человека в постановке, чтобы это не повторялось:**
- **Референс + правило ориентации курса** одним предложением («getRate читаемый >1, инвертируй через usdtPer») — снял бы №2/№4.
- **«Показывай всё/не фильтруй» и «не добавляй дефолты/фильтры сверх задачи»** — снял бы №5.
- **Ссылка на источник истины** (страница Tolunay, DDL сервера) в задаче — снял бы №3.
- **Явный критерий готовности с визуальной проверкой** («покажу скрин, тогда готово») — снял бы №1.

*Вывод для меня:* перед новой математикой/маппингом — сначала grep существующих хелперов и открыть источник; не отгружать визуальное без проверки; не добавлять фильтры/дефолты, которых не просили.
